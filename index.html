// SIEM & Log Management

The foundation of detection. Without comprehensive, quality logging you are flying blind. Invest in coverage before tooling.

Log Source Priority Matrix

TIER 1 — CRITICAL (must have)
├── Windows Security Event Log   4624/4625 (logon) · 4688 (process) · 4698 (sched task) · 4662 (AD obj access)
├── Windows Sysmon               Process · network · registry · file · pipe · DNS · WMI events
├── EDR Telemetry                Behavioral events · process injection · memory anomalies
├── Network Firewall / IDS        Allowed + denied connections · lateral movement
├── DNS Logs                      Query logs → C2 beacon detection · domain generation
└── Identity Provider (Entra/Okta) Sign-ins · MFA · Conditional Access · risk signals

TIER 2 — HIGH VALUE
├── Proxy / Web Filter            HTTP/HTTPS traffic · new domains · large uploads
├── Email Security Gateway        Phishing campaigns · attachment analysis · link clicks
├── VPN / Remote Access           Authentication · unusual hours · geoanomalies
├── Active Directory              Replication · group changes · LDAP queries · ADCS
├── PowerShell ScriptBlock (4104)  Script content logging · amsi data
└── Exchange / M365 Audit         Mailbox access · forwarding rules · consent grants

TIER 3 — SITUATIONAL
├── WMI Activity Log              WMI subscriptions · remote exec
├── NetFlow / IPFIX               East-west traffic baselines · beaconing
├── DHCP / RADIUS                 IP-to-hostname mapping · NAC bypass
├── Cloud API (CloudTrail / Defender) IAM changes · instance creation · exfil
└── Application / Web Server      WAF bypass attempts · web shell indicators

RETENTION GUIDANCE
├── Hot (searchable): 90 days minimum — 1 year preferred for enterprise
├── Warm (archived): 1–3 years — regulatory / legal hold requirements
└── Cold (tape): 3–7 years — compliance-dependent, restore on demand

Sysmon Deployment

Microsoft Sysinternals Sysmon — essential endpoint telemetry for Windows detection
  • Key event IDs: 1 (process), 3 (network), 7 (DLL load), 8 (CreateRemoteThread), 10 (ProcessAccess), 11 (file create), 12/13 (registry), 15 (ADS), 17/18 (pipe), 22 (DNS)
  • SwiftOnSecurity config: sysmonconfig.xml — widely used baseline, filter noisy processes first
  • Olaf Hartong's config: More comprehensive, modular — good for mature environments

  • Network events: Event 3 captures process-to-IP correlation — enables C2 beacon detection
  • DNS Event 22: Captures DNS queries with the requesting process — crucial for C2 over DNS detection
  • Performance: Filter aggressively — shipping all Sysmon events raw is expensive and noisy

// EDR & Endpoint Detection

Modern EDRs are the primary detection layer for endpoint attacks. Understanding what they see — and where they're blind — is essential for both detection tuning and incident response.

EDR Detection Layers — What Blue Team Sees

KERNEL CALLBACKS (highest fidelity — attacker cannot bypass)
├── PsSetCreateProcessNotifyRoutine  Every process creation → attacker can't hide new processes
├── PsSetCreateThreadNotifyRoutine   Remote thread creation → cross-process injection alert
├── PsSetLoadImageNotifyRoutine      DLL loads → image path, signed status, memory-backed check
└── ObRegisterCallbacks              Process handle access → LSASS access attempts

ETW TELEMETRY (rich but patchable in-process)
├── Microsoft-Windows-Threat-Intelligence  Kernel-sourced inject/read/write process events
├── .NET CLR ETW                             Assembly loads, JIT compilation
├── PowerShell ETW (4104)                    Script block content before execution
└── WMI-Activity Operational                 WMI provider loads and query execution

MEMORY SCANNING (periodic — gaps exist)
├── MZ/PE header in executable memory       Injected DLLs, reflective loaders
├── Unbacked executable regions             VirtualAlloc shellcode (no backing file)
├── Known shellcode signatures              CS/Sliver/Metasploit stagers
└── RWX memory regions                     Suspicious when combined with network activity

BEHAVIORAL ANALYTICS (ML + heuristics)
├── Call stack analysis     Detect shellcode with anomalous return addresses
├── Parent-child anomalies  Word spawning PowerShell, Office spawning cmd.exe
├── Credential access       LSASS reads, SAM access, DPAPI operations
└── Lateral movement        PsExec, WMI, service creation patterns across hosts

// Detection Engineering

Building, maintaining, and improving detection capabilities. Coverage over the MITRE ATT&CK framework, rule lifecycle management, and reducing false positive noise.

DETECT

SIGMA Rules

Vendor-agnostic detection rule format — write once, deploy anywhere
  • Format: YAML-based generic detection rules that compile to SIEM-specific query languages via sigma-cli
  • Rule components: logsource (category + product), detection (keywords/filters), falsepositives, level (informational → critical)
  • SigmaHQ: Community rule repository at github.com/SigmaHQ/sigma — 3000+ rules aligned to MITRE ATT&CK
  • Compilation: sigma convert -t splunk -p splunk_windows rule.yml — converts to platform-specific query
  • Tuning: Add filter conditions to suppress known false positives — filter_legit_software blocks
  • Rule lifecycle: Test in audit → measure FP rate → tune → enable → monitor regression
  • Detection-as-Code: Store rules in Git, CI/CD pipeline validates and deploys — version controlled detections
DETECT

YARA Rules

Pattern-based file and memory scanning — detect malware families by content
  • Strings section: Define byte patterns, text strings, or hex sequences to match
  • Condition: Boolean logic combining string matches with file properties (size, PE headers, entropy)
  • Modules: PE module for import tables, exports, sections; math module for entropy; dotnet for .NET metadata
  • Memory scanning: Scan live process memory with yara64 -p 4 rule.yar PID or via EDR integration
  • Threat intel YARA: Kaspersky, Mandiant, and Elastic publish free YARA rules for known malware families
  • Retrohunt: Upload rule to VirusTotal for retroactive file scanning across their corpus
  • Performance: Anchor rules with rare strings first; avoid broad conditions that scan everything
DETECT

MITRE ATT&CK Coverage Mapping

Map detection coverage to adversary TTPs — identify gaps before attackers exploit them
  • ATT&CK Navigator: Visual heatmap of technique coverage — color code by detection, prevention, or threat actor
  • Coverage tiers: Alert (detected), Logged (visible but no alert), Blind (no visibility) — know your gaps
  • Prioritization: Use threat intelligence to prioritize coverage for techniques used by actors targeting your sector
  • Detection testing: Atomic Red Team tests mapped 1:1 to ATT&CK — validate detections fire on real technique execution
  • Purple team: Red team executes techniques, blue team validates alert coverage — document results in ATT&CK Navigator
  • D3FEND: MITRE D3FEND maps countermeasures to ATT&CK techniques — use to identify defensive controls needed
DETECT

Detection Rule Development

Building high-fidelity, low-noise detections that analysts will trust and act on
  • Start with attack simulation: Execute the technique (Atomic Red Team / Caldera), observe what data sources fire, build rule from real data
  • Specificity vs. sensitivity: Broad rules catch more, high FP rate kills analyst trust. Narrow rules miss variants
  • Baseline first: Understand normal before writing anomaly rules — what does legit PsExec look like in your environment?
  • Layered detection: Single technique, multiple rule angles — process creation + network + registry = chained alert
  • False positive tracking: Log every FP with source — build suppression knowledge base over time
  • Detection retirement: Rules that never fire in 6 months need review — either gap in logging or detection is broken
DETECT

Behavioral Analytics & UEBA

User and Entity Behavior Analytics — detect anomalies when signatures won't help
  • Baseline period: Collect 30-90 days of normal behavior before enabling anomaly alerts — shorter = more noise
  • Key use cases: Impossible travel, off-hours access, privileged account misuse, data staging before exfil
  • Peer group analysis: Compare user behavior to their peer group — developer accessing finance systems is anomalous
  • Entity scoring: Aggregate risk signals per user/host over rolling window — alert on cumulative score exceeding threshold
  • ML model drift: Normal behavior changes over time (new role, project) — periodic retraining or supervised feedback loops
  • Tools: Splunk UBA, Microsoft Sentinel UEBA, Securonix, Exabeam, Elastic ML anomaly detection
DETECT

Network Detection & Response (NDR)

Network-layer visibility for detecting C2, lateral movement, and data exfiltration
  • Zeek (Bro): Network security monitor generating connection logs, HTTP, DNS, SSL, Kerberos, SMB metadata
  • Suricata: IDS/IPS with PCAP analysis — ET Open rules cover C2 frameworks, exploit kits, malware families
  • Arkime (Moloch): Full PCAP capture and indexing — retrospective packet analysis when you need raw evidence
  • JA3/JA3S fingerprinting: TLS client/server fingerprints — identify C2 frameworks by TLS hello characteristics
  • JARM: Active TLS server fingerprinting — scan internal hosts to identify C2 team servers with known signatures
  • Beaconing detection: Statistical analysis of connection interval regularity — C2 sleep+jitter creates detectable patterns
  • DNS analytics: Long subdomain strings, high-entropy domains, unusual query types (TXT, MX for C2)

// Incident Response

Structured response process from initial alert to full recovery. Speed, documentation, and scope-accuracy determine outcomes. Contain early, understand fully before eradicating.

IR Lifecycle & Key Decision Points

PHASE 1: PREPARATION
├── IR Runbooks: Documented procedures per incident type (ransomware, BEC, insider threat)
├── Contact lists: Legal, PR, executive, cyber insurance, law enforcement — pre-agreed escalation paths
├── Asset inventory: Know your crown jewels before an incident — what matters most?
└── Tabletop exercises: Run scenarios quarterly — muscle memory reduces response time

PHASE 2: DETECTION & ANALYSIS
├── Initial triage: True positive confirmation · scope estimation · severity classification
├── Timeline construction: Patient zero → first action → lateral spread → impact
├── Avoid tunnel vision: First compromised host ≠ only compromised host
└── Preserve evidence: Memory capture BEFORE rebooting; log preservation before rotation

PHASE 3: CONTAINMENT
├── Short-term: Network isolate affected hosts — EDR isolation or VLAN change
├── Account containment: Disable compromised accounts, revoke tokens, kill active sessions
├── Don't tip off attacker: Silent containment vs. visible response — attacker may accelerate if they notice
└── Scope before you act: Containing too early on wrong host = attacker still active elsewhere

PHASE 4: ERADICATION
├── Remove all persistence: Scheduled tasks · services · registry keys · DLL sideloads · WMI subs
├── Clean or reimage: For sophisticated attackers, reimaging > cleaning
├── Credential reset scope: EVERY credential that could have been exposed must be rotated
└── AD-wide compromise: Reset krbtgt TWICE (48hr apart) · all DA accounts · DPAPI backup key

PHASE 5: RECOVERY & LESSONS LEARNED
├── Validate before reconnecting: Confirm clean before returning to production
├── Monitor closely post-recovery: Attacker may have left secondary persistence
├── Root cause analysis: What control failed? Why? What will prevent recurrence?
└── Detection updates: Every IR should generate at least one new detection rule
IR

Triage & Scope Assessment

Rapidly determine the true scope of compromise — don't eradicate before you understand the full picture
  • Patient zero identification: Work backwards from the alert — what was the first malicious action on this host?
  • Lateral movement mapping: Use authentication logs (4624/4648) to trace account usage across all hosts
  • Credential scope: What accounts logged into the compromised host? All are potentially compromised
  • Data touched: What file shares, databases, email were accessed? Determines breach notification scope
  • Tool: Velociraptor — hunt for attacker artifacts across the entire fleet in minutes
  • Timeline tool: Plaso/log2timeline — super-timeline from multiple forensic artifacts for accurate chronology
  • Key question: Is the attacker still active? Monitor without tipping off while scoping
IR

Memory Forensics

Capture and analyze volatile memory — find injected code, C2 connections, and in-memory credentials
  • Capture tools: winpmem (Windows), LiME (Linux kernel module), osxpmem (macOS)
  • Analysis framework: Volatility 3 — plugins for process listing, network connections, injected code, registry
  • Key Volatility plugins: pstree, netscan, cmdline, malfind (injected code), handles, dlllist
  • malfind: Identifies memory regions with MZ headers, executable without backing file, or RWX — primary injected code finder
  • Credential extraction: pypykatz on memory dump — extracts LSASS credentials for scope assessment
  • MemProcFS: Mount memory image as filesystem — browse process memory, extract files, search strings
  • Critical: Capture memory BEFORE rebooting or isolating — shutdown destroys volatile evidence
IR

Disk & Artifact Forensics

Persistent artifacts on disk — even after malware deletion, forensic traces remain
  • Master File Table (MFT): Timestamps of every file operation — find deleted malware by $MFT entry even after deletion
  • Prefetch: C:\Windows\Prefetch\ — records program execution with last-run timestamp and files accessed
  • ShimCache / AppCompatCache: Registry record of every EXE run on the system — even if deleted from disk
  • AmCache: C:\Windows\appcompat\Programs\Amcache.hve — SHA1 hash of recently executed binaries
  • Event Log analysis: EvtxECmd + Timeline Explorer (EricZimmerman tools) — parse and timeline event logs
  • Recycle Bin: Deleted files with original path and deletion timestamp in $I and $R files
  • LNK files / Jump Lists: Recent file access history — reveal what the attacker opened or staged
IR

Active Directory Incident Response

AD-wide compromise response — credential reset order, golden ticket invalidation, and Kerberos hardening
  • Confirm scope first: Use BloodHound on current AD snapshot to map what the attacker could reach from compromised accounts
  • krbtgt rotation: MUST rotate TWICE with ≥10 hours between rotations — single rotation does not invalidate all golden tickets
  • DA reset order: Reset Domain Admins, then Enterprise Admins, then built-in Administrator — in that order
  • DPAPI domain backup key: If attacker had DCSync, they have the backup key — all user secrets encrypted with it are compromised
  • Trust relationships: Review all domain trusts — attacker may have added rogue trusts for persistent access
  • ADCS audit: Check for newly issued certificates, template modifications, new CA installations post-compromise
  • Tool: PingCastle — quick AD health and security posture assessment post-IR
IR

Cloud IR — Entra ID & M365

Responding to cloud identity compromise — token revocation, app audit, and CAP hardening
  • Revoke all sessions: Revoke-MgUserSignInSession — kills all active tokens including refresh tokens
  • App registration audit: Check for new app registrations and consent grants since compromise window — attacker may have registered backdoor app
  • OAuth consent grants: Review all delegated permissions — attacker abuses consent for persistent mail/file access that survives password reset
  • Unified Audit Log: Enable if not already — captures sign-ins, mail access, SharePoint activity. Requires E3/E5 license
  • Microsoft Entra ID Protection: Review risk detections — atypical travel, anonymous IP, malware-linked — correlate with your incident timeline
  • Conditional Access gaps: Audit CAPs for exclusions, legacy auth allowances, and unprotected apps — close the path used by attacker
  • Passkey/FIDO2 audit: Check for newly enrolled authenticators — attacker may have registered their own MFA device
IR

Ransomware-Specific Response

Ransomware IR is distinct — speed, backup integrity, and legal considerations are paramount
  • Immediate actions: Network segment (not full shutdown) — preserve forensic evidence while preventing spread
  • Identify strain: ID Ransomware (nomoreransom.org) — upload note/encrypted file to identify variant, check for decryptor
  • Backup validation: Test backup integrity IMMEDIATELY — ransomware groups often target backups first (Veeam, shadow copies)
  • Do NOT pay without legal advice: OFAC sanctions may apply to some ransomware groups — paying may be illegal
  • Pre-encryption activity: Look for data exfiltration before encryption — "double extortion" means backup recovery doesn't prevent leaks
  • Patient zero: Usually phishing or exposed RDP — trace to initial access for remediation
  • Cyber insurance: Notify immediately — policies have strict notification windows, non-compliance voids coverage

// Threat Hunting

Proactive hunting for adversaries that evaded automated detection. Hypothesis-driven investigation using knowledge of adversary TTPs to find what rules missed.

HUNT

Hunt Methodology

Structured approach to proactive threat hunting — intelligence-driven, hypothesis-based, data-validated
  • 1. Intelligence input: What TTPs are actors targeting your sector using? Use CTI reports, ISAC alerts, recent IR findings
  • 2. Hypothesis formation: "If APT29 is in our environment, we would expect to see OAuth token theft patterns in sign-in logs"
  • 3. Data collection: Identify which log sources support the hypothesis — ensure they're being collected
  • 4. Hunt execution: Run queries, analyze results, pivot on findings. Document every pivot
  • 5. New detection: Every hunt should produce either a finding OR a new detection rule for the pattern investigated
  • Frequency: Weekly structured hunts aligned to threat intel; ad-hoc after major CVE or industry breach
  • Tool: ThreatHunter-Playbook (OTRF) — community hunt playbooks with query templates
HUNT

Hunt: C2 Beaconing

Detect regular callback patterns that signature-based rules miss
  • Beaconing signature: Connections to same destination at regular intervals — even with jitter, statistical patterns persist
  • RITA (Real Intelligence Threat Analytics): Automated beaconing detection via connection regularity scoring
  • Zeek conn.log analysis: Group by src/dst pair, calculate interval deviation — low deviation = beaconing
  • Long-connection hunting: HTTP connections >60s with low byte count = potential interactive C2
  • JA3 hunting: Correlate JA3 fingerprints against known C2 tool signatures (CS default JA3 is well-known)
  • DNS beaconing: Regular low-TTL queries to same domain — even if domain resolves, pattern is suspicious
  • Splunk hunt: | eventstats count, avg(bytes) as avg_bytes by src, dest | where count>10 AND avg_bytes<1000
HUNT

Hunt: Living Off the Land

Find attacker abuse of legitimate Windows tools and built-in binaries
  • LOLBin baseline: Know which LOLBins are legitimately used in your environment — rundll32 calling COM objects is normal; calling URLs is not
  • PowerShell hunting: Encoded commands (-enc flag), download cradles (Net.WebClient, Invoke-Expression), bypass flags
  • mshta / wscript / cscript: Unusual parent processes or network connections from these binaries
  • certutil: Network connections from certutil.exe = high confidence IOA for download activity
  • Process lineage: Office → PowerShell, Explorer → cmd → PowerShell are high-risk parent-child chains
  • WMIC: Remote WMI execution creates WmiPrvSE.exe child processes on target — hunt parent-child chains from WMI
  • MSBuild / InstallUtil: Spawning network connections or executing shellcode in .NET context
HUNT

Hunt: Credential Theft

Detect LSASS access, Kerberoasting, and credential-based lateral movement
  • LSASS access (Event 10): GrantedAccess flags 0x1FFFFF, 0x1010, 0x0410 = dump attempts. Exclude known AV/EDR process names
  • Kerberoasting: Event 4769 with ticket encryption type 0x17 (RC4) for service accounts — should be AES only
  • Password spraying: Multiple 4625 failures from same source across many accounts in short window
  • DPAPI abuse: Unusual process accessing CryptProtectData / CryptUnprotectData on another user's DPAPI keys
  • Credential manager: Rundll32 keymgr.dll or vaultcmd.exe with no user context = automated credential harvesting
  • DCSync: Event 4662 with replicating directory changes right + source is not a DC IP/hostname
  • AS-REP Roasting: Event 4768 with RC4 pre-auth disabled accounts — check for unusual service ticket requests
HUNT

Hunt: Persistence Mechanisms

Find attacker-installed persistence that survived initial incident response
  • Autoruns baseline: Collect Autoruns output across fleet with AutorunsToWinEventLog — diff against known-good baseline
  • Scheduled task hunting: New tasks, tasks with encoded PowerShell in actions, tasks with unusual authors or paths
  • Service hunting: Services with non-standard paths, services pointing to user-writable directories, new services
  • Registry Run keys: Sysmon Event 13 monitoring HKCU/HKLM Run keys — especially from user context
  • WMI subscriptions: Query Get-WMIObject __FilterToConsumerBinding — any subscription not in your baseline is suspicious
  • COM hijacking: HKCU CLSIDs that shadow HKLM entries — especially InprocServer32 pointing to user-writable paths
  • DLL sideloads: Process Monitor "NAME NOT FOUND" entries for DLLs in writable paths — especially from new processes
HUNT

Atomic Red Team

Open-source library of adversary technique tests — validate detection coverage empirically
  • Repository: github.com/redcanaryco/atomic-red-team — 1000+ tests mapped to MITRE ATT&CK
  • Invoke-AtomicRedTeam: PowerShell framework to execute atomics and verify prerequisites
  • Workflow: Run atomic → check SIEM for alert → if no alert, fix detection → document coverage
  • Caldera: MITRE's adversary emulation platform — chain atomics into full attack scenarios with autonomous agents
  • Stratus Red Team: Cloud-specific ATT&CK tests for AWS, Azure, GCP — validate cloud detection coverage
  • Purple team cadence: Monthly atomic runs on key techniques; quarterly full chain emulations with red team
  • Safe execution: Run in dedicated test environment or use cleanup commands — some atomics are destructive

// Identity & Access Security

Identity is the new perimeter. Every credential compromise bypasses network controls. Hardening identity systems is the highest-ROI defensive investment for most organizations.

IDENTITY

Multi-Factor Authentication

MFA tiers — not all MFA is equal. FIDO2 defeats AitM; TOTP does not.
  • FIDO2 / Passkeys (phishing-resistant): Hardware token (YubiKey) or device-bound passkey — cryptographically bound to domain, defeats AitM/EvilGinx
  • Windows Hello for Business: Device-bound phishing-resistant credential — preferred for Windows endpoints
  • Certificate-based Authentication: Smart card / virtual smart card — phishing-resistant, device-bound
  • TOTP / Authenticator app (vulnerable to AitM): Easily bypassed by EvilGinx-style reverse proxy — better than nothing
  • SMS / voice (weakest): SIM swap, SS7 attacks, social engineering — use only as last resort fallback
  • MFA fatigue: Attackers spam push notifications until user accepts — enable number matching and additional context
  • Priority targets: Enable phishing-resistant MFA for all privileged accounts first — then enforce broadly via Conditional Access
IDENTITY

Privileged Access Management

Protect, monitor, and govern privileged accounts — the keys to your kingdom
  • Administrative tiering: Tier 0 (DC/CA/PKI), Tier 1 (servers), Tier 2 (workstations) — strict credential isolation between tiers
  • Privileged Access Workstations (PAW): Dedicated hardened devices for admin tasks — no email, no browsing, no untrusted software
  • Just-In-Time (JIT) access: PIM/Azure Privileged Identity Management — elevate only when needed, time-boxed, requires justification
  • Just-Enough-Access (JEA): PowerShell constrained runspaces limiting admin to specific cmdlets — no interactive shell
  • Password vaulting: CyberArk, BeyondTrust, Delinea — rotate privileged passwords automatically, session recording
  • LAPS (Local Admin Password Solution): Unique local admin password per machine — stops lateral movement via shared local creds
  • Protected Users group: Prevents Kerberos delegation, NTLM auth, and credential caching for high-priv accounts
IDENTITY

Active Directory Hardening

Reduce the AD attack surface — the primary target in 90% of enterprise breaches
  • Disable legacy authentication: Block NTLM where possible; force Kerberos. Monitor for RC4 ticket requests
  • AdminSDHolder: Audit members of protected groups — any unexpected account with AdminCount=1 is suspicious
  • Kerberos delegation: Audit unconstrained delegation (golden ticket amplifier) — restrict or eliminate where possible
  • Account hygiene: Disable unused accounts, clean up stale service accounts, enforce password policies
  • Kerb pre-authentication: Ensure ALL accounts have pre-auth required — AS-REP roastable accounts are an easy win for attackers
  • ADCS hardening: Run Locksmith — fix ESC1-ESC8 misconfigurations, restrict template enrollment rights
  • Audit LDAP queries: Enable LDAP diagnostic logging to detect enumeration — BloodHound-style queries generate distinctive patterns
IDENTITY

Conditional Access Policies

Zero trust access control — enforce context-based policies for every authentication
  • Baseline policies: Require MFA for all users; block legacy authentication; require compliant/hybrid-joined devices for sensitive apps
  • Named locations: Define trusted IPs for relaxed policies; enforce stricter controls from unknown locations
  • Sign-in risk: Integrate Entra ID Protection risk signals — block high-risk sign-ins, require step-up auth for medium risk
  • Device compliance: Require Intune-managed and compliant device for access to sensitive resources
  • Phishing-resistant MFA: Enforce FIDO2/WHfB for all admin roles, VPN access, and sensitive applications
  • Audit exclusions: Every CAP exclusion is a potential bypass — minimize, document, and review exclusions quarterly
  • Report-only mode: Test CAPs in report-only before enforcing — measure impact on sign-ins before blocking users
IDENTITY

Entra ID / Azure AD Security

Cloud identity hardening for Microsoft 365 and Azure environments
  • Global Admin minimization: Maximum 5 Global Admins, all with FIDO2 hardware keys, dedicated cloud-only accounts
  • PIM for all privileged roles: No permanent role assignments for privileged roles — all require PIM activation with justification
  • App consent policies: Restrict user consent — require admin approval for apps requesting sensitive permissions
  • External collaboration: Restrict guest access, limit B2B invitations, audit existing guest accounts quarterly
  • Secure Score: Microsoft Secure Score — track security posture and prioritize hardening actions
  • Graph activity logging: Enable MicrosoftGraphActivityLogs for post-compromise investigation — default not enabled
  • Emergency access accounts: Two break-glass accounts with FIDO2 keys — stored separately, excluded from all CAPs, monitored for use
IDENTITY

Password Security

Password policy, breach monitoring, and resistance to credential attacks
  • Banned password lists: Microsoft Entra Password Protection — blocks commonly used passwords and custom banned terms (company name, seasons, years)
  • Credential breach monitoring: Have I Been Pwned API — check new/changed passwords against known breach databases
  • Password length > complexity: NIST 800-63B recommends long passphrases over complex-but-short passwords
  • Eliminate periodic rotation: NIST guidance — only rotate on confirmed compromise, not arbitrary schedule. Rotation breeds predictable patterns
  • Password manager policy: Mandate password manager use — stops reuse, enables long random passwords
  • Spray detection: Alert on 5+ distinct users with failed auth from same IP within 1 hour — typical spray pattern

// Network Security Controls

Network segmentation, traffic inspection, and protocol-level controls. Every east-west movement the attacker makes should cross a control point you monitor.

NETWORK

Network Segmentation

Limit blast radius through logical and physical network separation
  • VLAN segmentation: User, server, management, guest, IoT — each segment with appropriate firewall rules between
  • Micro-segmentation: Host-based firewall policies that allow only needed application flows — stops east-west movement even within same VLAN
  • Crown jewel isolation: DCs, CA servers, backup infrastructure — separate VLAN with strict ACLs, jump host access only
  • Zero Trust Network Access (ZTNA): Application-layer access control — verify user, device, and context before permitting each connection
  • Deny by default: Whitelist required flows, deny all else — not the common "deny what's known bad" approach
  • Egress filtering: Outbound connections should be explicitly allowed — not default-permit. Block direct-to-internet from servers
  • SMB firewall: Block TCP 445 between workstations — prevents PsExec lateral movement without legitimate admin need
NETWORK

DNS Security

DNS is used in nearly every attack — logging, filtering, and monitoring are essential
  • DNS logging: Log all DNS queries with requesting process name (Sysmon Event 22) — enables C2 and DGA detection
  • DNS sinkholes: Redirect malicious domains to internal sinkhole IP — log and alert on connections, block C2 callback
  • Protective DNS: Cisco Umbrella, Cloudflare Gateway, Quad9 — filter malicious domains at resolver level
  • DNS-over-HTTPS (DoH) control: Block or proxy DoH — attackers use DoH to bypass DNS monitoring (Cloudflare, Google)
  • Response Policy Zones (RPZ): BIND/Infoblox — block or redirect by domain list at DNS server level
  • DGA detection: High-entropy domain names, NX domain storms — indicators of malware beaconing with domain generation algorithm
  • Internal DNS consistency: Use internal DNS for all lookups — prevents split-brain DNS attacks
NETWORK

Email Security Hardening

Technical controls to reduce phishing delivery success rates
  • SPF: Publish strict SPF record — v=spf1 include:... -all (hard fail). Reject unauthorized senders
  • DKIM: Cryptographically sign all outbound email — validates sender authenticity, prevents header tampering
  • DMARC: v=DMARC1; p=reject; rua=mailto:... — enforce and receive aggregate reports on spoofing attempts
  • BIMI: Brand Indicators for Message Identification — verified logo display in clients signals legitimate email
  • Attachment sandboxing: Detonate attachments before delivery — catch macro-less payloads, LNK files, ISO files
  • URL rewriting + time-of-click scanning: Rescan URLs when clicked — catches URLs that were clean at delivery
  • Anti-spoofing: Block emails from external that impersonate internal domains — common BEC vector
NETWORK

Web Proxy & SSL Inspection

Inspect encrypted traffic and control web access to detect and block C2 channels
  • SSL/TLS inspection: Decrypt HTTPS traffic for malware scanning, DLP, and C2 detection — required to inspect modern attacks
  • URL categorization: Block uncategorized, newly-registered, and parked domains — attackers use these for C2
  • Application control: Block unsanctioned cloud storage, anonymizers, and proxies — DLP and C2 channel control
  • User-Agent filtering: Alert or block curl, python-requests, Invoke-WebRequest user-agents from non-dev systems
  • Domain age checks: Alert on connections to domains registered less than 30 days ago — freshly-registered C2 infrastructure
  • JA3 fingerprinting at proxy: Detect known C2 frameworks by TLS client hello fingerprint
  • Egress proxy enforcement: Force all traffic through proxy — block direct internet TCP 80/443 from endpoints
NETWORK

Firewall & IDS/IPS

Perimeter and internal controls for traffic filtering and intrusion detection
  • Suricata / Snort: Network IDS/IPS — Emerging Threats (ET) Open rules cover C2 frameworks, exploit kits, malware callbacks
  • Next-Gen Firewall (NGFW): Palo Alto, Fortinet, CheckPoint — application-aware rules, SSL inspection, threat prevention
  • Internal IDS placement: Span ports at core switch — east-west traffic visibility for lateral movement detection
  • Geo-blocking: Block inbound from countries with no business justification — reduces attack surface for credential stuffing
  • ET Ruleset tuning: Enable, test, then tune ET rules — ET Pro provides faster updates and more coverage than Open
  • Alert on new C2 frameworks: When new C2 tools are released publicly (Havoc, Brute Ratel), add specific detection rules
NETWORK

VPN & Remote Access Security

Secure remote access as an initial access vector — heavily targeted by threat actors
  • MFA enforcement: All VPN must require MFA — single-factor VPN is direct internal access for credential stuffers
  • Certificate-based auth: Client certificate pinned to device — prevents credential spray from attacker-controlled devices
  • Split tunneling policy: All traffic through VPN in high-security environments — prevents attacker using victim's VPN tunnel while browsing attacker infrastructure
  • VPN authentication logging: Log all success AND failure events — baseline normal usage patterns
  • Geo-anomaly detection: Alert on VPN logins from countries the user never uses — rapid geographic changes are impossible travel
  • Session limits: Concurrent session limits — one active session per user, new login terminates old session

// Security Hardening

Reduce attack surface through configuration management, policy enforcement, and defense-in-depth controls. Every unnecessary feature is potential attack surface.

HARDEN

Windows Endpoint Hardening

CIS benchmark implementation, attack surface reduction, and LOLBin restriction
  • Attack Surface Reduction (ASR) rules: Enable all 16 ASR rules in audit first — then block mode. Key rules: block Office spawning processes, block macro downloads, block credential theft from LSASS
  • Credential Guard: Hypervisor-protected LSASS — prevents pass-the-hash and DPAPI attacks against domain credentials
  • Device Guard / WDAC: Kernel-mode code integrity — only signed drivers, strong code integrity policy. Kills BYOVD driver attacks
  • PowerShell Constrained Language Mode: Restrict PS to safe subset — block Add-Type, New-Object, and reflection
  • Script Block Logging: Enable 4104 — captures all PS script content, even from memory (ConvertTo-SecureString etc.)
  • AppLocker: Enforce application whitelisting — block execution from user-writable paths (AppData, Temp, Downloads)
  • Disable macros: Block all VBA macros or restrict to digitally signed only — Group Policy controls via Trust Center settings
HARDEN

Windows Defender Configuration

Maximize built-in Windows Defender capabilities — it's more powerful than most realize
  • Tamper Protection: Prevents local modification of Defender settings — enforce via Intune or Microsoft Endpoint Manager
  • Cloud-delivered protection: Enable at highest blocking level — real-time ML cloud scoring for unknown samples
  • Automatic sample submission: Enable — submits suspicious samples to Microsoft for analysis
  • Network protection: Enable in block mode — prevents connections to malicious domains/IPs at kernel level
  • Controlled Folder Access: Protects common data directories from unauthorized write access — ransomware prevention
  • Exploit protection: Enable ASLR (mandatory), CFG, DEP, SEHOP for all processes — process-level exploit mitigations
  • Safe Links / Safe Attachments: MDE for Office — detonate attachments and rewrite URLs in real time
HARDEN

ADCS Hardening

Active Directory Certificate Services is one of the most dangerous misconfiguration areas in enterprise AD
  • Locksmith: Run Locksmith — identifies ESC1-ESC8 vulnerabilities in certificate templates automatically
  • ESC1 fix: Remove "Supply in the Request" (ENROLLEE_SUPPLIES_SUBJECT) from non-CA templates — prevents SAN abuse
  • ESC4 fix: Remove write permissions on certificate templates for non-admin users — prevents template modification
  • CA audit logging: Enable all CA audit events — certificate issuance, template changes, CA start/stop
  • Restrict enrollment agents: Limit who can enroll on behalf of others — enrollment agent abuse enables impersonation
  • ADCS Tier 0: CA servers are Tier 0 assets — same access controls as domain controllers
  • Certificate monitoring: Alert on certificates issued with SAN containing privileged account UPNs, unusual templates, or off-hours issuance
HARDEN

LAPS (Local Admin Password Solution)

Unique local admin passwords per machine — stops lateral movement via shared local credentials
  • Windows LAPS (2023+): Built-in OS feature replacing legacy LAPS — Azure AD and on-prem AD support, encrypted storage
  • Scope: All domain-joined Windows computers — workstations AND servers
  • Password complexity: Configure 20+ character password with full character set
  • Rotation frequency: Rotate after each admin session, or at minimum every 30 days
  • Access audit: Log all LAPS password retrievals — bulk retrieval is suspicious (attacker sweeping for lateral movement)
  • Tiered access: Restrict who can read LAPS passwords — not all helpdesk needs server LAPS passwords
  • Why it matters: Without LAPS, a single shared local admin password lets attackers PtH to every machine in the domain
HARDEN

Patch Management

Vulnerability remediation prioritized by exploitability, exposure, and asset criticality
  • CISA KEV: Known Exploited Vulnerabilities catalog — treat these as P0, patch within 2 weeks regardless of CVSS score
  • Exploitation probability: EPSS score (Exploit Prediction Scoring System) — prioritizes by likelihood of exploitation, not just severity
  • Asset criticality: Internet-facing + Tier 0 = immediate patching. Internal + low criticality = standard SLA
  • Emergency patching: Pre-approved change process for critical CVEs — can't wait for monthly maintenance window when CVSS 9.8 is being exploited
  • Patch validation: Test in staging, confirm in production — especially for Windows security updates that ship with known issues
  • Third-party software: Browsers, PDF readers, Office suites — often more targeted than OS. Automate via Intune/SCCM
HARDEN

Backup & Recovery Hardening

Ransomware-resistant backup strategy — your last line of defense when all else fails
  • 3-2-1-1-0 rule: 3 copies, 2 different media, 1 offsite, 1 immutable/offline, 0 errors confirmed by restore testing
  • Immutable backups: Object lock on S3 / Azure Blob — prevents ransomware from deleting or encrypting backups
  • Offline backups: Air-gapped tapes or offline disks — cannot be reached by ransomware spreading through network
  • Backup credential isolation: Backup service accounts in dedicated OU with no regular access — attackers target Veeam/Commvault to destroy backups
  • Recovery testing: Quarterly restore tests — validate RTO/RPO before you need it. Most orgs discover backup failures during incidents
  • DR runbook: Documented step-by-step recovery procedure — under stress, detailed runbooks reduce errors and time-to-recover

// Deception Technology

Honeypots, canary tokens, and deceptive credentials give high-fidelity alerts with near-zero false positives. Any interaction is malicious by definition.

DECEPTION

Canary Tokens

Tripwires embedded in documents, credentials, and files — immediate alert on access
  • Thinkst Canary Tokens: Free at canarytokens.org — Word docs, PDFs, web bugs, AWS keys, DNS tokens, email addresses
  • Word/Excel tokens: Embed in sensitive documents — fires when opened, reveals opener IP and hostname
  • Fake AWS keys: Place in source code, S3 buckets, configuration files — fires on any API call attempt
  • DNS tokens: Fake hostnames in internal documentation — fires on DNS lookup, reveals attacker has read internal docs
  • Honey credentials: Fake accounts in AD with canary passwords — login attempt is instant attacker detection
  • Desktop folder canaries: Fake "passwords.xlsx" and "network_map.pdf" on shared drives — opened = attacker browsing shares
  • Detection value: Near-zero false positive rate — legitimate users don't open fake documents or use fake credentials
DECEPTION

Honeypots

Fake systems that attract and detect attackers conducting internal reconnaissance
  • Thinkst Canary: Commercial hardware/virtual honeypots — realistic Windows, Linux, web, and network device emulation. Alert on any interaction
  • OpenCanary: Free, open-source honeypot — simulates SSH, HTTP, FTP, SNMP, SMB, and other services
  • HoneyAD: Fake AD accounts, computers, and service accounts — activity against honeypot objects = attacker in AD
  • Placement strategy: Scatter across VLANs, mimic real naming conventions, no legitimate reason to connect
  • Honey credentials in LSASS: Inject fake credentials that call home when used — detects credential harvesting from memory
  • Network deception platforms: Attivo (SentinelOne), Illusive — automated deception fabric across the entire network
DECEPTION

Active Directory Deception

Honey accounts, honey SPNs, and deceptive AD objects that detect enumeration and attacks
  • Honey users: Fake privileged-looking accounts (svc_backup, IT_Admin) — any auth attempt is an alert
  • Honey computer objects: Fake DC or server names in AD — Kerberoasting attempt against them reveals attacker
  • Honey SPNs: Service Principal Names on honey accounts — Kerberoast attempt reveals enumeration activity
  • Breadcrumb strategy: Leave hints pointing to honey systems — fake "password file" referencing honey server name
  • Honey group members: Put honey accounts in sensitive groups — any lateral movement attempt to them alerts
  • Monitoring requirement: Honey objects only work if you alert on all authentication attempts against them — zero tolerance for false positives
DECEPTION

Canary Files & Data Deception

Bait files that trigger alerts when accessed — detect exfiltration attempts and insider threats
  • DLP canary seeding: Place realistic-looking fake sensitive files in monitored shares — access triggers DLP alert AND canary token
  • Fake credit card data: Seeded Luhn-valid but non-real card numbers in files — DLP alerts on the data pattern
  • Honey API keys: Valid-format but revoked API keys — any attempt to use them fires an alert via AWS/Azure logging
  • Fake password databases: KeePass-format files with canary tokens inside — opened = attacker found credential files
  • Cloud storage canaries: Files in S3 with server-access logging + canary token — fire on any read access

Detecting C2 Frameworks

DETECT

Detecting ADCS Attacks

Certificate-based attacks are difficult to detect — dedicated logging and monitoring required
  • Enable CA audit logging: Event 4886 (cert issued), 4887 (cert issued with SAN), 4897 (role separation) — not enabled by default
  • Alert on SAN in certs: Any certificate issued with a Subject Alternative Name containing a privileged UPN (Domain Admins, admin accounts)
  • Off-hours certificate issuance: Certificates requested/issued outside business hours for privileged accounts
  • Template modification: Event 4899 — certificate template modified. Should be rare and change-controlled
  • PKINIT anomalies: Kerberos pre-auth using certificate from account that never used cert auth before
  • ESC detection rules: Sigma rules for ADCS exploitation available in SigmaHQ — alert on ESC1-ESC8 exploitation patterns
DETECT

Detecting Credential Attacks

Comprehensive detection for the full credential theft and abuse lifecycle
  • Password spray: Event 4625 — 5+ unique accounts failing auth from single source in 60 minutes
  • Kerberoasting: Event 4769, Ticket Encryption Type = 0x17 (RC4), from non-service account. Alert on bulk requests
  • AS-REP Roasting: Event 4768 without pre-authentication for accounts that should have it enabled
  • PtH: Event 4624 Type 3 NTLM with no corresponding 4648 — NTLM network logon without explicit credential use
  • DCSync: Event 4662 with Object Type = Domain, Access Mask = 0x100, from non-DC. Add exclusion for MSOL sync accounts
  • DPAPI abuse: Unusual process calling CryptUnprotectData for another user's data — check process integrity and parent
  • Browser credential theft: Unusual process accessing SQLite databases in Chrome/Edge AppData paths
DETECT

Detecting Email-Based Attacks

Phishing detection, AitM patterns, and email-based initial access vectors
  • EvilGinx detection: Impossible travel immediately after login (different IP for the cookie vs. login); browser fingerprint mismatch
  • Device code phishing: Sign-in from new device immediately after device code completion from different IP; alert on rapid device code usage
  • OAuth consent abuse: New app registration granted mail.read or files.read permissions by non-admin users
  • Inbox rule creation: Forwarding rules to external domains, deletion rules matching keywords — common in BEC
  • MailSniper patterns: Bulk EWS access, search operations across many mailboxes from single account — data exfil pattern
  • Email impersonation: Sender domain visually similar to trusted domain — alert on lookalike domains via DMARC reports

// SOC Operations

Running an effective Security Operations Center — from alert handling and shift handover to metrics, tooling, and analyst development.

SOC

Alert Triage Workflow

Structured approach to processing alerts — minimize MTTD and MTTR while managing analyst load
  • Triage SLA: All Critical alerts acknowledged in 15 min; High in 1 hour; Medium in 4 hours. Track against SLA in SOAR
  • Enrichment automation: Auto-enrich every alert with user details, asset owner, threat intel lookups, prior incidents — before analyst sees it
  • Priority queues: Separate queues by severity and data source — don't mix P1 EDR alerts with P3 vulnerability findings
  • Context gathering: Before escalating, analyst should answer: What happened? When? On what? By whom? Is it still happening?
  • False positive management: Track every FP. After 3 FPs from same rule, tune or suppress. Build a suppression knowledge base
  • Escalation criteria: Pre-defined criteria for L1→L2→IR escalation — removes judgment calls in high-stress moments
SOC

SOAR Automation

Automate repetitive response actions — let analysts focus on judgment, not mechanics
  • Enrichment playbooks: Auto-run VirusTotal, Shodan, WHOIS, internal asset lookup on every IOC — analysts see enriched context immediately
  • Containment playbooks: Auto-isolate endpoint on critical EDR alert; auto-disable account on critical identity alert (with approval workflow)
  • Notification playbooks: Auto-notify asset owner on first critical alert; escalate to IR manager if no acknowledgment in SLA window
  • Ticket creation: Auto-create JIRA/ServiceNow ticket with full context for every true positive — no manual data entry
  • Platforms: Palo Alto XSOAR, Splunk SOAR, Microsoft Sentinel Playbooks (Logic Apps), TheHive + Cortex, Shuffle (open source)
  • Human in the loop: Automate enrichment and notification freely; gate destructive actions (block, disable, isolate) behind analyst approval
SOC

SOC Metrics

Measuring SOC effectiveness — operational health, detection quality, and response performance
  • MTTD: Mean Time to Detect — from attacker first action to alert generation. Target: <24 hours
  • MTTR: Mean Time to Respond — from alert to containment action. Target: <4 hours for critical
  • MTTC: Mean Time to Contain — from detection to full containment. Target varies by severity
  • Alert-to-incident ratio: What % of alerts become confirmed incidents? Too high = missing coverage; too low = too much noise
  • False positive rate per rule: Track by detection rule — high FP rules need tuning or retirement
  • ATT&CK coverage %: What % of techniques have at least one detection? Track quarterly improvement
  • Dwell time: How long attackers were present before detection — the primary security outcome metric
SOC

Case Management & Documentation

Track incidents, preserve evidence chains, and build institutional knowledge
  • TheHive: Open-source security incident response platform — case management, tasks, observables, and Cortex analyzer integration
  • IRIS: Open-source IR case management — timeline, graph relationships, evidence tracking, report generation
  • Jira / ServiceNow: Enterprise ticketing with customized IR workflows — links to SIEM alerts, tracks SLA, escalation paths
  • Timeline documentation: Every IR case needs a chronological timeline — attacker actions with timestamps, analyst actions, containment events
  • Chain of custody: Document evidence collection steps — who collected, when, from where, hash of collected artifacts
  • Post-IR report: Executive summary + technical timeline + root cause + recommendations — produced for every major incident

├── 4648  Explicit credentials logon  │ runas / PtH preparation
├── 4672  Special privileges assigned  │ Admin logon · track DA usage

├── 4720  Account created              │ Unauthorized user creation
├── 4728  Member added to global group  │ Sensitive group (Domain Admins) membership change
├── 4756  Member added to universal group │ Enterprise Admins or high-priv group change
├── 4738  User account changed           │ Pre-auth disabled = AS-REP roastable; PwdNotReqd flag
└── 4765  SID History added              │ Privilege escalation via SID History injection

├── 4688  Process creation (with command line) │ Requires Audit + command line policy

├── 4768  Kerberos AS ticket            │ AS-REP roasting: pre-auth disabled accounts
